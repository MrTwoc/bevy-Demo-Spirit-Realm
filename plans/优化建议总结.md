# SuperChunk 必要性评估与优化路线图

> 本文档总结对 SuperChunk 合批方案的必要性评估，以及按收益排序的优化建议。
> 
> **结论**：SuperChunk 不是当前阶段的必要优化，存在架构缺陷，且存在更优先的改进方向。

---

## 一、SuperChunk 必要性评估

### 1.1 当前系统性能基线

| 指标 | 数值 |
|------|------|
| Chunk 实体数 | ~2,237 |
| Draw Call | ~2,200+ |
| FPS 稳态 | ~170 |
| FPS 峰值 | ~247 |
| FPS 谷值（加载尖峰）| ~58 |
| 帧时间 | ~5.9ms（均值），~21.8ms（峰值）|
| **瓶颈定位** | **CPU 端 Draw Call 提交** |

### 1.2 SuperChunk 预期收益

| 指标 | 当前 | SuperChunk 后 |
|------|------|---------------|
| 实体数 | ~2,237 | ~4 |
| Draw Call | ~2,200+ | ~4 |
| FPS 稳态 | ~170 | 预计 200+ |

理论上 Draw Call 降低 ~550x，FPS 提升约 18%。

### 1.3 SuperChunk 方案的结构性缺陷

#### Y 尺寸不匹配

- **当前 Y_LOAD_RADIUS=4**：玩家上下各加载 4 层 SubChunk（共 9 层）
- **方案 SuperChunk Y 尺寸：4 层**

这是结构性不匹配 — 9 层无法被 4 层整齐覆盖，SuperChunk 边界与玩家活动范围产生复杂交互。

#### 刚性合并触发条件

方案要求"所有 SubChunk 已加载完成才合并"，但世界边缘的 SubChunk 永远不会加载：
- 对应 SuperChunk 将永久卡在 `Pending` 状态，**整个区域无法渲染**
- "部分合并策略"和"超时回退"只是方向性建议，**没有具体实现方案**

#### 分帧合并复杂性被低估

修改单个方块后触发整个 SuperChunk（256 个 SubChunk）网格重建：
- 当前：修改方块 → 重建 ~2,000 triangles
- SuperChunk：修改方块 → 重建 ~500,000 triangles（256x）

方案声称"每帧合并 16 个 SubChunk，16 帧完成"是**猜测性估算**，新旧网格切换的一致性问题未设计。

### 1.4 被忽视的更重要问题

| 问题 | 优先级 | 影响 |
|------|--------|------|
| Atlas mipmap 边缘渗色 | P1 | 纹理质量 |
| SuperChunk 网格上传开销 | P1 | CPU 性能 |
| 远景 MegaLOD 瓦片更新机制缺失 | P1 | 玩家建造无法同步到远景 |

这些问题在 SuperChunk 方案中完全未涉及，但直接影响最终用户体验。

### 1.5 结论

> **SuperChunk 不是当前阶段的必要优化。**
> 
> 当前系统在 170 FPS 下运行良好，加载尖峰问题已有分帧加载机制缓解。SuperChunk 的架构缺陷（Y 尺寸不匹配、刚性合并条件）需要在方案设计阶段解决，而非实现阶段打补丁。

## 二、优化路线图（按收益/代价比排序）

### 2.1 优化方向收益/代价矩阵

| 优先级 | 优化方向 | 预估收益 | 代价 | 风险 |
|--------|----------|----------|------|------|
| **立即** | 分帧加载调参 | 消除加载尖峰 | 极低 | 极低 |
| **立即** | Atlas UV 收缩 | 修复边缘渗色 | 极低 | 极低 |
| **短期** | 异步网格生成 | 帧时间稳定性 | 中 | 中 |
| **中期** | LOD 系统（近景）| GPU 负载降 30-50% | 中 | 低 |
| **长期** | SuperChunk | CPU 开销大幅降低 | 高 | 高（需先修复架构）|
| **长期** | GPU 面提取 | CPU 开销大幅降低 | 高 | 高 |
| **长期** | MultiDrawIndirect | Draw Call → 1 | 极高 | 高 |

### 2.2 立即可做的改进

#### 分帧加载参数调优（消除加载尖峰）

当前配置导致加载时帧时间峰值 ~21.8ms，是均值 ~5.9ms 的 3.7 倍。

**建议调整**（[`src/chunk_manager.rs`](src/chunk_manager.rs:22-27)）：

```rust
// 当前：
CHUNKS_PER_FRAME = 4
LRU_UNLOADS_PER_FRAME = 16

// 建议：
CHUNKS_PER_FRAME = 2      // 减少加载速率，平滑曲线
LRU_UNLOADS_PER_FRAME = 8  // 减少卸载抖动
```

**预期效果**：加载尖峰从 ~21ms 降至 ~12ms，FPS 谷值从 ~58 提升至 ~80。

#### Atlas UV 收缩（修复纹理边缘渗色）

在 [`src/chunk.rs`](src/chunk.rs:480) 的 `face_quad` 中，向内收缩 UV 坐标 1-2px，防止双线性插值渗色。

```rust
// 建议（收缩约 0.5px / 32px）：
let eps = 0.016;
let face_uvs = [
    [u_min + eps, v_max - eps],
    [u_max - eps, v_max - eps],
    [u_max - eps, v_min + eps],
    [u_min + eps, v_min + eps],
];
```

#### 摄像机初始位置微调

当前摄像机在 (16, 64, 16)，地形基准高度 16，最高地形约 48。

建议改为 (16, 40, 16)，更贴近地表，减少初始空旷感。

## 三、后续任务规划

### 3.1 短期任务：异步网格生成

**目标**：将 `generate_chunk_mesh` 移出主线程，消除加载尖峰。

**简化方案**（不需要完整 GPU Compute Shader）：
1. 使用 `AsyncComputeTaskPool`（Bevy 内置）
2. 每帧将若干 SubChunk 的网格生成任务提交到后台
3. 主线程仅做 mesh 上传

**预估收益**：主线程帧时间降低 ~20-30%（网格生成约占 10-15%），加载期间 FPS 更稳定。

### 3.2 中期任务：LOD 系统

**目标**：减少远景面数，降低 GPU 负载。

**实施建议**（参考 [`docs/架构总纲.md`](docs/架构总纲.md:143-161)）：
- 0~4 区块：LOD0（全精度）
- 5~8 区块：LOD1（1:2 降采样）
- 9~16 区块：LOD2（1:4 降采样）

### 3.3 长期任务：SuperChunk（需先修复架构）

在实施 SuperChunk 前，必须先完成以下设计：

1. **重新设计 SuperChunk Y 尺寸**：调整为 9 层以匹配 Y_LOAD_RADIUS=4，或调整 Y 轴加载策略
2. **实现部分合并机制**：设计缺失 SubChunk 时的渲染策略（如用透明/空网格填充）
3. **设计分帧合并一致性**：旧网格保留、新旧切换、数据同步的完整方案
4. **评估脏标记扩散代价**：修改方块后整个 SuperChunk 重建，面数 256x，是否可接受

### 3.4 更长期：GPU 面提取 + MultiDrawIndirect

面向最终目标架构（128+ 区块可视距离）：

| 任务 | 说明 | 预估收益 |
|------|------|----------|
| GPU 面提取 | 将面剔除计算移出 CPU | CPU 开销大幅降低 |
| MultiDrawIndirect | 一次 Draw Call 渲染所有可见几何体 | Draw Call → 1 |

---

## 四、总结

| 阶段 | 任务 | 预期收益 |
|------|------|----------|
| **立即** | 分帧加载参数调优 + UV 收缩 | 消除加载尖峰、修复渗色 |
| **短期** | 异步网格生成 | 帧时间稳定性 |
| **中期** | LOD 系统 | GPU 负载降低 |
| **长期** | SuperChunk（需先修复架构设计）| CPU 开销大幅降低 |

> 建议先做"立即"阶段的两个改进，这些都是几小时内可完成、零风险的优化。
