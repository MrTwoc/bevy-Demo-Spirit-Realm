# 极致性能体素引擎设计方案（最终版：GPU 加速全栈）
**目标硬件**：GTX 1060 6GB + 同期四核 CPU（如 i5-7500）  
**目标帧率**：≥ 144 FPS @ 1080p/1440p（动态分辨率保障）  
**世界规格**：X/Z 轴无限延伸，Y 轴建筑高度 ±20480，区块大小 32×32×32  
**核心机制**：动态近景（可交互）+ 静态远景（LoD 只读）完全解耦  
**视距设定**：动态半径 12~16 区块（384~512 米），静态 LoD 半径 128 区块（4096 米），可扩展至 2048  
**关键技术栈**：Rust + Bevy + wgpu，WGSL 计算着色器用于网格生成、剔除与间接绘制

+ **动态近景**：玩家周围半径 12~16 区块（384~512 米）为全交互环境。
+ **静态远景**：自 17 区块起至 128 区块（4096 米）为预生成的极低多边形 MegaLOD 瓦片。
+ **独立渲染管线**：远景与近景在不同阶段渲染，利用深度缓冲自然合并。
+ **GPU 接管**：所有繁重计算（网格生成、可见性判别、绘制命令生成）均迁移到 GPU，CPU 仅负责任务调度和游戏逻辑。

## 2. 核心数据结构
### 2.1 区块数据 (`ChunkData`)
```rust
pub struct ChunkVoxels(pub Box<[u16; 32 * 32 * 32]>); // 64 KB 连续数组

pub enum ChunkData {
    Empty,                       // 全空气，零分配
    Uniform(u16),                // 全单一材质，仅 2 字节
    Mixed(ChunkVoxels),          // 异构区块，堆分配 64 KB
}
```

+ 坐标映射：`linear_index = x + y * 32 + z * 1024`
+ 内存占用极低：主视距内 5000~8000 区块，若全部 Mixed 也仅需 320~512 MB，结合 Uniform/Empty 实际远小于此值。
+ GPU 侧：64 KB 可直接以 `vec<u32>` 形式打包上传至 `StorageBuffer`，无需转换。

### 2.2 渲染数据结构
+ `SuperChunk`：覆盖 `8×8×4` 动态区块的合并网格实体，用于降低 Draw Call。
+ `MegaLODTile`：覆盖 `64×64×16` 原始区块的静态远景瓦片，每个瓦片对应一个极低多边形网格。
+ `GPUMeshCache`：驻留于显存的网格缓冲区池，避免重复分配。

## 3. 动态近景系统（0～12/16 区块）
### 3.1 交互管理
+ 仅在此范围内维护方块数据、物理、实体、AI。
+ 方块修改实时写入对应 `ChunkData` 数组，并标记脏 (`DirtyFlag`)。
+ 脏区块被调度到 GPU 计算队列以重新生成网格。

### 3.2 GPU 网格生成
**核心**：在 GPU 上一次性完成 32768 个体素的面提取与简单合并，彻底卸载 CPU。

#### 3.2.1 数据流
1. **上传**：脏区块的 64KB 数据通过 `Queue::write_buffer` 写入显存中的 `StorageBuffer`。
2. **面提取计算着色器**：
    - 每个线程处理一个体素，检查 6 个邻面。
    - 若相邻为空气/透明，则输出该面（4 个顶点 + 6 个索引）到 `AppendBuffer`。
    - 使用原子计数器分配输出位置。
3. **可选第二 pass**：沿轴向快速贪心合并四边形，减少顶点数（近景可跳过，远景必须做）。
4. **结果**：生成的顶点/索引数据可直接被 Bevy 的 `Mesh` 使用，无需 CPU 拷贝。

**性能**：单区块面提取约 0.05~0.1 ms（GTX 1060 GPU 时间），数千区块分摊到多帧，对帧率几乎无影响。

#### 3.2.2 WGSL 核心逻辑片段
```plain
@group(0) @binding(0) var<storage, read> voxels: array<u32>;
@group(0) @binding(1) var<storage, read_write> vertices: array<Vertex>;
@group(0) @binding(2) var<storage, read_write> indices: array<u32>;
@group(0) @binding(3) var<storage, read_write> face_count: atomic<u32>;

@compute @workgroup_size(256)
fn face_extract(@builtin(global_invocation_id) gid: vec3<u32>) {
    let idx = gid.x + gid.y * 32u + gid.z * 1024u;
    if (idx >= 32768u) { return; }
    let pos = vec3<i32>(gid);
    let mat = voxels[idx] & 0xFFFFu;
    if (mat == 0u) { return; } // AIR
    for (var dir = 0u; dir < 6u; dir++) {
        let npos = pos + DIRS[dir];
        if (!is_inside(npos) || get_mat(npos) == 0u) {
            let start = atomicAdd(&face_count, 4u);
            write_face(&vertices, &indices, start, pos, dir, mat);
        }
    }
}
```

### 3.3 SuperChunk 合批
+ 动态区块的渲染实体不再逐区块绘制，而是合并到覆盖 `8×8×4` 个区块的 `SuperChunk`。
+ 当内部任意区块网格更新，对应的 `SuperChunk` 标记脏，由后台 GPU 任务（或其他并行方式）重新合并所有子网格为一个单一 `Mesh`。
+ 视野内 SuperChunk 数量：以半径 16 区块为例，水平约 5×5 = 25 个，垂直 2 层 → **Draw Call ≈ 50**。
+ 尽管如此，下一步 GPU 剔除和 Indirect Draw 能将它压至更低。

### 3.4 近景 LOD 策略
+ 动态半径内一律使用 LOD0（全精度），避免不必要的 LOD 过渡开销。
+ 超过动态半径的区块立即转为静态远景管理。

## 4. 静态远景系统（17～128+ 区块）
### 4.1 MegaLOD 瓦片
关于——[关于——megaLOD](https://www.yuque.com/u1056894/xyw989/hg5kl8a58t9vorzl)

+ 世界被划分为 **64×64×16** 区块的固定瓦片（约 2048×2048×512 米）。
+ 每个瓦片预生成 **一个极低多边形网格**，代表该区域的宏观地形。
+ 根据瓦片中心到相机的距离，选择不同的网格精度：

| 距离范围（区块） | LoD 级别 | 采样步长 | 瓦片典型三角形数 |
| --- | --- | --- | --- |
| 17 – 32 | LOD1 | 4 | 5,000 – 20,000 |
| 33 – 64 | LOD2 | 8 | 1,000 – 5,000 |
| 65 – 128 | LOD3 | 16 | 200 – 1,000 |
| 129 – 2048 | LOD4 | 64 | < 500 |


+ **生成方式**：GPU 计算着色器离线（或背景低优先级）生成，序列化至磁盘，运行时直接加载 `Mesh` 资产。
+ **内存**：视野内瓦片数极少（128 视距时约 9~16 个），三角面总数 < 50 万，显存占用可忽略。

### 4.2 远景渲染
+ 远景瓦片在近景渲染之前绘制，利用深度缓冲自动合并。
+ 使用极简着色器：单一颜色/程序化纹理 + 远景雾效，无需法线/AO/实时光照。

### 4.3 垂直方向处理
+ 对于玩家上下距离极远的高度（如 y > 256 或 y < -64），瓦片内容自然为空或极简岩层外壳。
+ 高空和深地的封闭视觉由天空盒和雾效掩盖，不需要渲染实体。

## 5. GPU 驱动的剔除与间接绘制
### 5.1 可见性剔除（完全 GPU 侧）
1. **输入**：所有 `SuperChunk` 和 `MegaLODTile` 的包围盒（AABB），存于 `StorageBuffer`。
2. **视锥剔除**：计算着色器将包围盒与视锥体平面测试，标记可见实例。
3. **Hi-Z 遮挡剔除**：
    - 首先渲染远景低模到深度缓冲。
    - 构建深度金字塔。
    - 剔除着色器将当前实例的最小深度与金字塔对应层级比较，剔除被完全遮挡的物体。
4. **输出**：紧凑的可见实例列表及对应的 `DrawIndirect` 命令。

### 5.2 一次绘制调用（MultiDrawIndirect）
+ Bevy 通过自定义渲染阶段使用 `RenderPass::multi_draw_indirect`，直接提交所有可见实例。
+ **Draw Call 从 50+ 降低至 1**（所有静态近 / 远景几何体合并在同一绘制）。
+ 这彻底消除了 CPU 提交开销，保证最高帧率。

### 5.3 性能收益
+ CPU 每帧仅需更新少数动态对象的包围盒（如移动的实体），并发出一次绘制命令。
+ 剔除和绘制完全与游戏逻辑并行，GTX 1060 轻松应对。

---

## 6. 体素数据更新与 GPU 同步
+ 玩家修改方块时，CPU 直接通过映射缓冲区（`Buffer::slice`）或 `Queue::write_buffer` 更新 GPU 端对应 Chunk 的 64KB 数据。
+ 设置脏标志，调度该 Chunk 的网格重新生成。
+ 大规模操作（如爆炸、流体）可由专门的 compute shader 直接在 GPU 端修改多个区块的体素数组，形成纯 GPU 更新流水线。

## 7. Y 轴极限高度 (±20480) 的管理
+ **动态近景**：以玩家为中心，垂直加载 ±128 米（4 区块高度），其余区间一律 `ChunkData::Empty`。
+ **远景瓦片**：垂直方向只生成地表上下共 16 个区块（512 米）的内容，更深或更高的区域被跳过或用极薄的壳渲染。
+ 视觉上，玩家无法感知极限高度处的缺失，因为视野覆盖和雾效足够遮蔽。

## 10. 扩展路径：通往 2048 超视距
+ 远景 LOD4 瓦片进一步合并，将 8×8×2 个 LOD3 瓦片聚合成一个超低模单元。

+ GPU 剔除和间接绘制确保 Draw Call 仍为 1。

## 11. 总结
本方案通过 **Rust + Bevy + wgpu** 技术栈，融合了三项核心优化：

1. **解耦动态/静态世界**，按距离分配渲染资源。
2. **GPU 计算着色器** 接管网格生成、剔除与绘制，CPU 仅负责逻辑。
3. **MultiDrawIndirect + Hi‑Z 剔除** 将 Draw Call 降至极限，彻底释放现代 GPU 的吞吐能力。

在目标硬件 GTX 1060 上，方案可实现 **16 区块动态交互半径 + 128 区块远景视距 @ 144+ FPS**，并预留了扩展至 2048 视距的完备技术路径。这一设计在可维护性、性能和功能承载能力之间取得了最优平衡，为无限开放世界体素沙盒游戏提供了坚实的技术地基。

}

    chunk_distance(chunk_pos, player_pos) &lt; LOAD_RADIUS

class="ne-text">噪声生成 + 64KB 数据填充</span></p><p id="u41332240" class="ne-p"><span class="ne-text">后台线程 / GPU 异步，每帧预算可控</span></p><p id="uaf350ff9" class="ne-p"><span class="ne-text">网格生成</span></p><p id="uc5fcf9be" class="ne-p"><span class="ne-text">GPU 面提取（0.05~0.1ms/区块）</span></p><p id="u13fcb0f0" class="ne-p"><span class="ne-text">分摊到多帧，脏区块限帧生成</span></p><p id="ub367beaf" class="ne-p"><span class="ne-text">远景可见性</span></p><p id="u507e57be" class="ne-p"><span class="ne-text">切换 </span><code class="ne-code"><span class="ne-text">Visibility</span></code><span class="ne-text"> 组件</span></p><p id="u376f2849" class="ne-p"><span class="ne-text">Bevy 的 </span><code class="ne-code"><span class="ne-text">Visibility</span></code><span class="ne-text"> 系统开销极低</span></p><p id="ubf805c94" class="ne-p"><span class="ne-text">内存抖动</span></p><p id="ue760fd0b" class="ne-p"><span class="ne-text">分配/释放 64KB 连续数组</span></p><p id="u6fc6a1d0" class="ne-p"><span class="ne-text">可使用对象池预分配内存块</span></p><p id="u22d66346" class="ne-p"><span class="ne-text">总之，</span><strong><span class="ne-text">切换逻辑本质上就是一个廉价的“if 距离 &lt; 阈值”判断</span></strong><span class="ne-text">，真正的重活（数据生成、网格生成）都被架构隔离到了异步任务和 GPU 端，完全不干扰主线程的 144 FPS 节奏。</span></p><hr id="Slam9" class="ne-hr"><h2 id="JOFWM"><span class="ne-text">进阶：更平滑的混合（可选）</span></h2><p id="u332018ab" class="ne-p"><span class="ne-text">如果想彻底消除远景和近景切换时的视觉跳变（例如远处突然从模糊 LOD 变成清晰细节），可以在过渡区域让两者短暂共存，并对远景瓦片做一个淡入淡出或透明度混合。但这在体素游戏中通常并不必要，因为玩家注意力在移动和目标上，远景的骤然出现会被自然知觉忽略。</span></p><p id="uaf16e03f" class="ne-p"><span class="ne-text">你现在可以放心：</span><strong><span class="ne-text">切换是实时的，但简单到不会在性能图表上冒泡</span></strong><span class="ne-text">。如果有需要，我们还可以深入聊聊异步生成系统的具体调度细节。</span></p></details>
<details class="lake-collapse"><summary id="u6dff12c9"><span class="ne-text">玩家探索世界时生成的新区块，应该是近景范围内还是远景范围内？</span></summary><p id="uad1b4470" class="ne-p"><span class="ne-text">好问题！</span><strong><span class="ne-text">新探索的区块会先以远景身份生成，只有当它逼近玩家进入动态半径时，才会升级为近景。</span></strong></p><p id="u64f50da0" class="ne-p"><span class="ne-text">我们需要把“区块生成”拆成两个层次来理解：</span><strong><span class="ne-text">地形数据生成</span></strong><span class="ne-text"> 和 </span><strong><span class="ne-text">网格生成</span></strong><span class="ne-text">。它们在远近景的触发时机完全不同。</span></p><hr id="BChOK" class="ne-hr"><h2 id="Ixp2f"><span class="ne-text">1. 地形数据生成：在进入远景范围时就开始</span></h2><p id="ud66c9c51" class="ne-p"><span class="ne-text">当玩家向一个全新方向移动，最远处的远景瓦片（MegaLOD Tile）需要被显示。而瓦片必须知道自己覆盖的地形形状。</span></p><ul class="ne-ul"><li id="u93da403b" data-lake-index-type="0"><strong><span class="ne-text">触发时机</span></strong><span class="ne-text">：比静态远景可视距离再提前一些（例如远景半径 128 区块，实际可在 140 区块时离线生成）。</span></li><li id="udc5788d5" data-lake-index-type="0"><strong><span class="ne-text">生成内容</span></strong><span class="ne-text">：使用确定性噪声函数生成该区域的体素数据，但</span><strong><span class="ne-text">不会为每个 32³ 区块分配完整的 </span></strong><code class="ne-code"><span class="ne-text">ChunkData</span></code><span class="ne-text">。而是直接生成远景瓦片所需的</span><strong><span class="ne-text">粗采样高度图或稀疏体素表示</span></strong><span class="ne-text">，然后生成极低多边形网格。</span></li><li id="u4e7472f3" data-lake-index-type="0"><strong><span class="ne-text">执行位置</span></strong><span class="ne-text">：完全在后台线程（或 GPU 计算）中异步完成，优先级最低，绝不与近景争抢资源。</span></li><li id="ud08c9f8e" data-lake-index-type="0"><strong><span class="ne-text">结果</span></strong><span class="ne-text">：玩家看到的远方地平线是这些低模瓦片构成的静态地形。此时，该区域内的任何一个独立 32³ 区块都</span><strong><span class="ne-text">不占用内存、没有交互能力</span></strong><span class="ne-text">。</span></li></ul><hr id="j2c0S" class="ne-hr"><h2 id="WwosV"><span class="ne-text">2. 近景网格生成：进入动态半径时才激活</span></h2><p id="u26f60629" class="ne-p"><span class="ne-text">当玩家继续前进，某个远景区域被纳入动态加载半径（例如 ≤16 区块）时：</span></p><ul class="ne-ul"><li id="u9d7a8e8c" data-lake-index-type="0"><strong><span class="ne-text">触发时机</span></strong><span class="ne-text">：区块中心距玩家 &lt; </span><code class="ne-code"><span class="ne-text">LOAD_RADIUS</span></code><span class="ne-text">（16 区块）。</span></li><li id="u9270bca3" data-lake-index-type="0"><strong><span class="ne-text">生成内容</span></strong><span class="ne-text">：为该区域内的每个 32³ 区块分配完整的 </span><code class="ne-code"><span class="ne-text">ChunkData</span></code><span class="ne-text">（64KB 数组），用</span><strong><span class="ne-text">相同的噪声函数</span></strong><span class="ne-text">重新生成体素值（保证一致），然后通过 GPU 计算着色器生成高精度贪心网格。</span></li><li id="u0e0a644a" data-lake-index-type="0"><strong><span class="ne-text">执行位置</span></strong><span class="ne-text">：生成任务被调度到更高优先级的异步队列，分帧完成，确保玩家脚下的地形最快可用。</span></li><li id="ud8ec214b" data-lake-index-type="0"><strong><span class="ne-text">结果</span></strong><span class="ne-text">：玩家可见的高细节地形，且此时可以开始进行方块挖掘、搭建等交互。</span></li></ul><hr id="d1doA" class="ne-hr"><h2 id="xbzjD"><span class="ne-text">3. 为什么这样设计？</span></h2><p id="ud7e60855" class="ne-p"><span class="ne-text">设计原则</span></p><p id="ub67b990a" class="ne-p"><span class="ne-text">说明</span></p><p id="ub82c4a5a" class="ne-p"><strong><span class="ne-text">尽早显示</span></strong></p><p id="ue7b96b22" class="ne-p"><span class="ne-text">远景瓦片让玩家在任何方向上都能立刻看到世界轮廓，不会出现“世界在你眼前生成”的突兀感。</span></p><p id="uf20db9c8" class="ne-p"><strong><span class="ne-text">节省内存</span></strong></p><p id="u08dd0993" class="ne-p"><span class="ne-text">远景区块不分配 64KB 数组，只存一份瓦片网格。等到需要交互时才分配，避免上万区块撑爆内存。</span></p><p id="u04500055" class="ne-p"><strong><span class="ne-text">平滑过渡</span></strong></p><p id="u02709abb" class="ne-p"><span class="ne-text">近景生成时，远景瓦片仍短暂存在，直到动态网格覆盖。由于噪声生成结果一致，视觉上两套网格无缝衔接，玩家几乎感知不到切换。</span></p><p id="uc39fe25e" class="ne-p"><strong><span class="ne-text">性能可控</span></strong></p><p id="uc6975c37" class="ne-p"><span class="ne-text">远景瓦片生成是低频、大粒度的，一个瓦片覆盖 64×64×16 区块；近景区块生成是高频、小粒度的，但有严格步调与限帧，不冲击帧率。</span></p><hr id="Oo2c5" class="ne-hr"><h2 id="EVpPj"><span class="ne-text">4. 简化的生命周期图</span></h2><pre data-language="plain" id="iBdqp" class="ne-codeblock language-plain"><code>新区域进入远景视界
    ↓
[后台线程] 计算噪声 → 生成远景瓦片粗网格 → 保存/缓存
    ↓
区域进入动态加载半径
    ↓
[后台线程 + GPU] 生成 32³ ChunkData → 生成高精度网格 → 合并到 SuperChunk
    ↓
玩家交互修改方块
    ↓
[GPU] 重新生成该区块网格 → 更新 SuperChunk
    ↓
区域移出动态卸载半径
    ↓
释放 ChunkData，删除动态网格，远景瓦片继续显示（无需重新生成）</code></pre><hr id="Rz7Ob" class="ne-hr"><p id="ue3c38c2e" class="ne-p"><strong><span class="ne-text">总结：新探索的世界会先在远景范围内以“低模身份”降生，当玩家踏入它的领地，它才会获得“高模灵魂”和交互能力。</span></strong><span class="ne-text"> 这种分层次、按需生成的策略是支撑 128 区块视距与 144 FPS 的核心之一。</span></p></details>
<details class="lake-collapse"><summary id="u48d8548c"><span class="ne-text">区块之间相接又不接触空气的面，如何剔除</span></summary><p id="u372e0b5e" class="ne-p"><span class="ne-text">区块之间相邻面的剔除，是体素引擎里经典的“边界问题”。在动态近景中，我们采用 </span><strong><span class="ne-text">在 GPU 生成网格时直接跨块查询邻居数据</span></strong><span class="ne-text">，并辅以 </span><strong><span class="ne-text">脏标记扩散</span></strong><span class="ne-text"> 的机制来完美解决。下面拆解它的实现细节。</span></p><hr id="uBhCh" class="ne-hr"><h2 id="kBmoi"><span class="ne-text">1. 问题核心</span></h2><ul class="ne-ul"><li id="uc45864cb" data-lake-index-type="0"><span class="ne-text">你有一个 32³ 的区块 A，它的一个体素在 A 的边界上（比如 x=31）。</span></li><li id="ub2006934" data-lake-index-type="0"><span class="ne-text">这个体素紧挨着邻居区块 B 的 x=0 位置。</span></li><li id="u5c5c8d9d" data-lake-index-type="0"><span class="ne-text">如果 B 的对应体素也是固体，那么 A 的这个面应该被剔除（内部面）。</span></li><li id="u32a29aa9" data-lake-index-type="0"><span class="ne-text">如果 B 是空气、未加载或为空，则 A 的面保留（暴露在空气中）。</span></li></ul><p id="ucf3fb925" class="ne-p"><strong><span class="ne-text">挑战</span></strong><span class="ne-text">：GPU 计算着色器在处理 A 时，没有 B 的数据。需要一种高效的方式让着色器“看到”邻居。</span></p><hr id="El6nh" class="ne-hr"><h2 id="HjYr5"><span class="ne-text">2. 解决方案：打包邻居数据并行传入</span></h2><p id="u8df55734" class="ne-p"><span class="ne-text">在调度 GPU 为某个区块生成网格时，我们不仅上传它自己的 64KB 数据，</span><strong><span class="ne-text">还同时传入它 6 个相邻区块的 64KB 数据</span></strong><span class="ne-text">（总共 7 个 64KB 缓冲区）。如果某个邻居不存在（未加载或为空），则传入一个全空气的替代缓冲区。</span></p><h3 id="vMWHx"><span class="ne-text">2.1 上传策略</span></h3><pre data-language="rust" id="gHcX7" class="ne-codeblock language-rust"><code>// 伪代码：为区块A准备GPU任务

    } else {

    }

        local.y &gt;= 0 &amp;&amp; local.y &lt; 32 &amp;&amp;
        local.z &gt;= 0 &amp;&amp; local.z &lt; 32) {
        let idx = local.x + local.y * 32 + local.z * 1024;

    // 标记当前区块脏
    mark_dirty(chunk_pos);
    // 检查是否在边界（x=0或31, y=0或31, z=0或31）
    for (axis, value) in [(X, local_pos.x), (Y, local_pos.y), (Z, local_pos.z)] {
        if value == 0 {
            mark_dirty(chunk_pos - unit_vector(axis)); // 负方向的邻居
        }
        if value == 31 {
            mark_dirty(chunk_pos + unit_vector(axis)); // 正方向的邻居
        }
    }
}</code></pre><p id="u3a32c2b3" class="ne-p"><span class="ne-text">这样相邻区块会在后续几帧内更新网格，所有内部面都会保持正确。</span><strong><span class="ne-text">因为脏标记扩散是极低频率的（只有边界修改才触发），几乎不增加重建负担。</span></strong></p><hr id="A7DRV" class="ne-hr"><h2 id="dmOax"><span class="ne-text">4. 远景瓦片不需要处理</span></h2><p id="uc564a323" class="ne-p"><span class="ne-text">远景的 </span><code class="ne-code"><span class="ne-text">MegaLOD Tile</span></code><span class="ne-text"> 是覆盖极大区域（如 64×64×16 区块）的单个网格，</span><strong><span class="ne-text">内部不存在区块边界</span></strong><span class="ne-text">，因此所有内部面都在同一网格生成时被剔除，天生就没有这个问题。只有动态近景的 32³ 区块需要上述机制。</span></p><hr id="C7OiG" class="ne-hr"><h2 id="q18BW"><span class="ne-text">5. 为什么不用其他方案</span></h2><ul class="ne-ul"><li id="uec96baa5" data-lake-index-type="0"><strong><span class="ne-text">深度测试自动剔除</span></strong><span class="ne-text">：不剔除内部面，靠 Early‑Z 让 GPU 深度测试丢弃被遮挡的面。但这会白白浪费顶点处理带宽和光栅化资源，对于 500 万三角形可能变成 2000 万以上的负载，GTX 1060 吃不消。</span></li><li id="u8f404e68" data-lake-index-type="0"><strong><span class="ne-text">CPU 后处理剔除</span></strong><span class="ne-text">：需要 GPU 回读生成的网格数据，CPU 剔除后再上传，引入同步点和延迟，完全违背了我们“GPU 一步到位”的设计。</span></li><li id="ud600ae05" data-lake-index-type="0"><strong><span class="ne-text">全局唯一体素缓冲区</span></strong><span class="ne-text">：维护一个巨大的全局 GPU 体素存储，让所有区块共享。这会带来复杂的同步和地址映射，在小规模动态范围下性价比低。</span></li></ul><p id="u70f12f67" class="ne-p"><strong><span class="ne-text">传入邻居缓冲区的方案</span></strong><span class="ne-text">，既保留了 GPU 并行生成的纯净性，又用极小的额外数据量（448KB）完美解决了边界剔除，是我们当前架构下的最优解。</span></p><hr id="lhB9Z" class="ne-hr"><p id="ub91c56e3" class="ne-p"><strong><span class="ne-text">总结：在 GPU 生成网格时，每个区块都携带 6 个邻居的完整数据，使面提取能透明地跨区块查询材质。方块修改扩散脏标记来保证边界面的实时更新。这一设计完全消除了区块接缝处的内部面，且不会引入任何 CPU 阻塞或多余 Draw Call。</span></strong></p></details>
