## 三、性能生命周期分析

整个测试可划分为 **6 个阶段**：

### 阶段 1：初始加载（t=0 ~ 10s）

| 指标 | 起始 | 结束 | 趋势 |
|------|------|------|------|
| FPS | 167.8 | 56.2 | ↓ 急剧下降 |
| 帧时间 | 6.06ms | 17.79ms | ↑ 3x |
| 区块数 | 124 | 1,267 | ↑ 10x |
| 三角面 | 1.06M | 4.55M | ↑ 4.3x |

**分析**: 区块快速加载导致三角面数从 1M 暴涨到 4.5M，FPS 从 168 降至 56。此阶段异步网格生成正常工作（无明显尖峰），但 Draw Call 数量随区块线性增长是主要瓶颈。

### 阶段 2：中度加载（t=10 ~ 50s）

| 指标 | 起始 | 结束 | 趋势 |
|------|------|------|------|
| FPS | 56.2 | 12.7 | ↓ 持续下降 |
| 帧时间 | 17.79ms | 78.45ms | ↑ 4.4x |
| 区块数 | 1,267 | 2,991 | ↑ 2.4x |
| 三角面 | 4.55M | 7.36M | ↑ 1.6x |

**分析**: 区块数增长 2.4x 但三角面仅增长 1.6x，说明远处区块的体素密度较低。然而 FPS 从 56 降至 13，帧时间达 78ms，远超 16.67ms（60FPS）目标。**Draw Call 提交开销**已成为严重瓶颈。

### 阶段 3：重度加载（t=50 ~ 110s）

| 指标 | 起始 | 结束 | 趋势 |
|------|------|------|------|
| FPS | 230.0（瞬态） | 292.7 | 波动剧烈 |
| 帧时间 | 4.35ms | 3.46ms | 短暂恢复 |
| 区块数 | 3,079 | 4,151 | 趋于稳定 |
| 三角面 | 7.36M | 8.05M | 缓慢增长 |

**分析**: 此阶段出现多次 FPS 瞬态飙升（230~292 FPS），对应帧时间 3.4~4.5ms。这些尖峰出现在区块数停止增长的瞬间（t=46s、t=50s、t=110s），推测原因：
1. **LOD 系统生效** — 远距离区块降采样后三角面大幅减少
2. **GPU 管线空闲** — 区块加载暂停时 GPU 能快速处理积压的渲染任务
3. **缓存命中** — 连续帧渲染相同区块集时 GPU 缓存有效

### 阶段 4：稳态高负载探索（t=110 ~ 355s）⚠️ **最严重问题**

| 指标 | 最佳 | 最差 | 中位数 |
|------|------|------|--------|
| FPS | 18.3（t=108s） | **3.8**（t=242s） | ~5.5 |
| 帧时间 | 54.5ms | **267.8ms** | ~180ms |
| 区块数 | 4,151 | 7,536 | ~6,500 |
| 三角面 | 7.87M | 9.54M | ~9.4M |

**分析**: 这是整个测试中 **性能最差的阶段**。随着玩家移动探索，新区块持续加载（从 4,151 增长到 7,536），导致：
- **Draw Call 数量爆炸**: 7,536 个区块 = ~7,536 次 Draw Call，远超 GPU 批处理效率
- **三角面逼近千万**: 9.54M 三角面对 GPU 压力巨大
- **帧时间飙升至 267ms**: 仅 ~3.8 FPS，完全不可交互

关键数据点：
- t=178s: **7.8 FPS**, 128.7ms, 6,264 区块
- t=242s: **3.8 FPS**, 263.4ms, 6,825 区块 ← **最低点**
- t=228s: **5.0 FPS**, 200.9ms, 6,734 区块

### 阶段 5：卸载事件（t=355 ~ 358s）

| 指标 | t=356s | t=358s | 变化 |
|------|--------|--------|------|
| FPS | 6.1 | 30.8 | ↑ 5x |
| 帧时间 | 162.8ms | 32.5ms | ↓ 5x |
| 区块数 | 7,536 | 5,090 | ↓ 32% |
| 三角面 | 9.54M | 8.57M | ↓ 10% |

**分析**: 区块从 7,536 骤降至 5,090（LRU 淘汰或切换区域触发卸载），FPS 立即从 6 恢复到 31。这证实 **区块数量（Draw Call 数量）是当前首要瓶颈**。

### 阶段 6：卸载后恢复态（t=358 ~ 402s）

| 指标 | 最佳 | 最差 | 中位数 |
|------|------|------|--------|
| FPS | 244.3（t=385s） | 49.7（t=398s） | ~65 |
| 帧时间 | 4.14ms | 20.1ms | ~15ms |
| 区块数 | 5,090 | 5,090 | 稳定 |
| 三角面 | 8.68M | 9.06M | ~9.06M |

**分析**: 区块数稳定在 5,090 后，FPS 恢复到 65~244 范围。但后半段（t=393~399s）FPS 再次降至 50~65，说明即使区块数不变，**三角面从 8.68M 增长到 9.06M 仍会显著影响性能**。

## 四、核心瓶颈诊断

### 🔴 瓶颈 1：Draw Call 数量线性增长（最严重）

| 区块数 | Draw Call（估算） | 帧时间 |
|--------|-------------------|--------|
| 4,151 | ~4,151 | ~3.5ms |
| 5,090 | ~5,090 | ~5ms |
| 6,500 | ~6,500 | ~150ms |
| 7,536 | ~7,536 | ~165ms |

**结论**: 每增加 1,000 个区块，帧时间增加约 40~50ms。Draw Call 提交的 CPU 开销与区块数呈**超线性关系**（可能是驱动层排序或状态切换开销）。

**这是 Phase 2（GPU 面剔除 + MultiDrawIndirect）的核心驱动力。**

### 🟡 瓶颈 2：三角面总量过高

- 稳定态三角面: 8.7M ~ 9.5M
- 文档目标: GPU 三角面在 LOD 后应显著降低

**当前 LOD 系统虽已启用，但三角面降幅有限**（从 ~9.5M 降到 ~8.7M，仅降 8%）。可能原因：
1. LOD3（1:8 降采样）的区块数量占比不足
2. LOD 降采样算法仍生成过多三角面
3. 近处 LOD0 区块的三角面过于密集

### 🟡 瓶颈 3：三角面 GPU 计数缺失

CSV 中 `triangle_count_gpu` 列始终为 **N/A**，无法对比 CPU 端和 GPU 端实际渲染的三角面数。这阻碍了以下分析：
- GPU 剔除是否生效
- 面可见性检查的剔除率
- 实际渲染负载与理论负载的差距

### 🟢 已验证优化效果

| 优化项 | 效果 | 证据 |
|--------|------|------|
| **Phase 0 异步网格生成** | ✅ 加载尖峰已消除 | 加载期间无帧时间突刺（对比基线 21.8ms） |
| **LOD 系统** | ⚠️ 部分生效 | t=110s 瞬态 FPS 292，但长期探索时效果有限 |
| **LRU 缓存淘汰** | ✅ 有效 | t=358s 卸载后 FPS 立即恢复 |

## 六、优化建议优先级

基于本次性能基准数据，重新评估优化路线：

### P0 — 紧急：Draw Call 合批（MultiDrawIndirect）

| 项目 | 详情 |
|------|------|
| **问题** | 7,536 个 Draw Call，CPU 提交开销 ~150ms |
| **目标** | Draw Call 合并至 <100 次 |
| **方案** | Phase 2 MultiDrawIndirect，将同材质区块合并为单次 Draw |
| **预期收益** | 帧时间从 150ms 降至 <10ms |
| **实现难度** | 🔴 高（需 wgpu 直接调用） |

### P1 — 高优：LOD 三角面优化

| 项目 | 详情 |
|------|------|
| **问题** | LOD 后三角面仅降 8%（9.5M → 8.7M） |
| **目标** | LOD 后三角面降至 <5M |
| **方案** | 1. 优化 LOD 降采样算法，减少冗余面<br>2. 增加 LOD3 覆盖范围<br>3. 实现跨 LOD 接缝面剔除 |
| **预期收益** | GPU 渲染负载降低 40-50% |

### P1 — 高优：启用 GPU 三角面计数

| 项目 | 详情 |
|------|------|
| **问题** | `triangle_count_gpu` 始终为 N/A |
| **方案** | 在 [`perf_logger.rs`](../src/perf_logger.rs) 中读取 GPU 侧诊断数据 |
| **预期收益** | 获得真实 GPU 渲染负载数据，指导后续优化 |

### P2 — 中优：GPU 面剔除（Compute Shader）

| 项目 | 详情 |
|------|------|
| **问题** | CPU 逐面检查开销随区块数线性增长 |
| **方案** | Phase 2 GPU Compute Shader 并行面剔除 |
| **依赖** | MultiDrawIndirect 基础设施 |
| **预期收益** | CPU 面剔除从 ~0.5ms/chunk 降至 <0.1ms/chunk |

### P3 — 低优：视距动态调整

| 项目 | 详情 |
|------|------|
| **问题** | 固定 RENDER_DISTANCE=32 导致远处区块过多 |
| **方案** | 根据当前 FPS 动态调整渲染距离（60FPS 时缩减，120FPS 时扩展） |
| **预期收益** | 自适应性能，避免极端低帧率 |

## 八、结论

1. **当前系统在 32 区块视距下性能严重不足**，稳态 FPS 仅 5~15，远低于 60FPS 目标
2. **Draw Call 线性增长是首要瓶颈**，7,500+ 次 Draw Call 导致 CPU 提交开销达 150ms+
3. **LOD 系统已部分生效**，在区块数稳定时可将 FPS 提升至 270~300，但无法抵消持续加载的压力
4. **Phase 0 异步网格生成成功消除了加载尖峰**，但未解决 Draw Call 数量问题
5. **下一步必须实现 MultiDrawIndirect**（Phase 2），将 Draw Call 合并至 <100 次，这是从 5 FPS 回到 60+ FPS 的关键路径
6. **GPU 三角面计数缺失**阻碍了精确的性能分析，建议优先修复
