# Spirit Realm 架构总纲

> 整合体素管理方案 + SVO 空间管理 + 动态 Atlas，生成统一架构文档。
> 项目目标：Y 轴 ±20480，XZ 无限延伸，可视距离 128+ 区块，比 Voxy 更远。

---

## 1. 项目规模

```
Y轴范围：-20480 ~ +20480（共 40960 格，是原版 MC 256 层的 160 倍）
XZ范围：无限延伸
可视距离：128+ 区块（约 32x128=4096 米），目标 2048 区块
参考：Voxy MOD（Minecraft 128+ 区块视距）
```

**核心矛盾**：
- 原版 MC 320 层，32³ chunk 够用
- 40960 层垂直方向，如果也用 32³ chunk，Y 方向需要 1280 个 chunk/列，内存和调度复杂度爆炸
- 必须引入**垂直分层流式加载**机制

## 3. 区块分层设计

### 3.1 垂直分层（Y 轴解决方案）

40960 层垂直方向，采用三级分层：

```
World Column（世界列）
└── MegaColumn（垂直大列）
    ├── SubChunk（32³）—— Y方向每32格一个
    └── 数量：40960 / 32 = 1280 个 SubChunk/列
```

| 层级 | 尺寸 | 用途 |
|------|------|------|
| SubChunk | 32³ | 最小加载单位（当前方案）|
| SuperChunk | 8×8×4 SubChunk | 近景合批单位（256×256×128 米）|
| MegaLOD Tile | 64×64×16 SubChunk | 远景瓦片（2048×2048×512 米）|

> **备用方案：16×32×16 SubChunk**
>
> 当未来区块尺寸32³无法实现 Y=±20480 超大垂直范围时，将 XZ 从 32 缩小为 16，
> 减少空气体素浪费。迁移成本低，不影响后续任何方案（SuperChunk、LOD、GPU 管线等）。
> 详见 [SubChunk 尺寸分析](#311-subchunk-尺寸分析)。

#### 3.1.1 SubChunk 尺寸分析

**当前选择 32³ 的理由**：

- 代码简洁：单一 `CHUNK_SIZE` 常量，无需区分 X/Y/Z
- Draw Call 更少：同样视距下区块实体数更少（约 1/4）
- 边界检查更少：跨区块面剔除频率更低
- SuperChunk 合批不受影响：任何尺寸都能做 8×8×4 合批

**16×32×16 的优势（备用）**：

- 内存效率更高：每 SubChunk 8192 体素 vs 32768，空气区块浪费更少
- 加载粒度更细：移动时只需加载更小的区块
- Y 轴适配更好：XZ 面积更小，减少无关空气体素

### 3.2 空间索引（XZ 无限延伸）

XZ 方向无限，用 HashMap 做稀疏索引：

```rust
// 区块坐标（SubChunk 级）
struct ChunkCoord {
    cx: i32,  // 每单位 = 32 格
    cy: i32,  // 每单位 = 32 格
    cz: i32,  // 每单位 = 32 格
}

// 全局 SubChunk 存储
struct ChunkManager {
    chunks: HashMap<ChunkCoord, ChunkEntry>,
    // LRU 淘汰：离视点越远越先卸载
    lru_cache: LinkedHashMap<ChunkCoord, u64>,
}
```

- 键：(cx, cy, cz)，值：ChunkEntry
- 只有加载过的区域才分配内存
- LRU 淘汰：高速移动时超出缓存上限的区块被强制卸载

### 3.3 ChunkData 三态存储

每个 SubChunk 内部用三态压缩：

```rust
pub enum ChunkData {
    Empty,                        // 全空气，0 字节
    Uniform(u16),                 // 全同一种方块，2 字节
    Mixed(PalettedChunkData),     // 调色板压缩，~4KB（32³=32768体素）
}
```

- 32³ = 32768 体素，调色板压缩后约 ~4KB/区块
- 三态存储让 90%+ 的空气/均匀区块几乎零内存

## 4. LOD 分级系统

### 4.1 近景（动态区域）

| 距离 | LOD | 精度 | 典型三角面/区块 |
|------|-----|------|----------------|
| 0~4 区块 | LOD0 | 1:1（SubChunk 原生）| ~2000 |
| 5~8 区块 | LOD1 | 1:2（8³ 采样）| ~500 |
| 9~16 区块 | LOD2 | 1:4（16³ 采样）| ~125 |

**注意**：近景（0~16 区块）全启用动态交互，网格随玩家移动异步重建。

### 4.2 远景（静态区域）

| 距离 | LOD | 采样 | 典型三角面/瓦片 | 数量/视距128 |
|------|-----|------|----------------|-------------|
| 17~32 区块 | LOD1 | 4 | 5K~20K | ~12 |
| 33~64 区块 | LOD2 | 8 | 1K~5K | ~12 |
| 65~128 区块 | LOD3 | 16 | 200~1K | ~8 |
| 129~2048 区块 | LOD4 | 64 | <500 | 递减 |

**远景预生成**：MegaLOD Tile（64×64×16 SubChunk）在后台线程或 GPU 预生成，序列化缓存，运行时直接加载 Mesh。

### 4.3 Y 轴流式加载

```
玩家在 Y=500 的位置：
├── 垂直加载范围：±128 米（如 Y=372~628）
├── 1280 个 SubChunk/列中，只加载 ~16 个（±4 个 Y 层）
├── 更高/更低的 SubChunk → Empty 或 Uniform（空气）
```

## 5. 渲染管线

### 5.1 GPU 视锥剔除 + Hi-Z 遮挡剔除

```
输入：所有 SuperChunk / MegaLOD Tile 的 AABB
     ↓
GPU Compute：视锥 6 平面测试
     ↓
Hi-Z Pass：深度金字塔，逐层剔除被遮挡物体
     ↓
输出：可见实例列表 + DrawIndirect 命令
     ↓
Bevy RenderPass：一次提交所有可见几何体
```

**性能收益**：Draw Call 从 50+ 降至 **1**（MultiDrawIndirect）。

### 5.2 跨区块面剔除（邻居数据打包）

SubChunk 面提取时，需要知道 6 个邻居的材质。解决方案：

```rust
// 调度 GPU mesh 生成时，同时上传：
// · 当前区块数据（~2KB）
// · 6 个邻居数据（~12KB）
// · 如果邻居不存在，传全空气替代缓冲区

dispatch_gpu_mesh_gen(current_data, neighbor_data[6]);
```

WGSL 着色器中跨区块查询：

```wgsl
fn get_voxel_world(pos: vec3<i32>) -> u32 {
    // 查当前区块 → 查邻居缓冲区 → 全空气
}
```

### 5.3 方块修改 → 脏标记扩散

```
玩家放置/破坏方块：
    ↓
修改 ChunkData（如需要，从 Uniform → Mixed）
    ↓
标记当前 SubChunk 为脏
    ↓
如果在边界（local pos = 0 或 31），同时标记 6 个邻居为脏
    ↓
调度 GPU 重新生成受影响 SubChunk 的网格
    ↓
更新 SuperChunk（重新合批）
```

## 8. 总结

| 维度 | 方案 |
|------|------|
| 空间索引 | HashMap + LRU（稀疏，无限 XZ）|
| SubChunk 尺寸 | 32³（当前方案）；16×32×16 作为备用方案 |
| 数据存储 | ChunkData 三态（Empty/Uniform/Mixed）|
| 近景合批 | SuperChunk（8×8×4）|
| 远景瓦片 | MegaLOD Tile（64×64×16）|
| LOD | 近景 LOD0~2 / 远景 LOD1~4 |
| 渲染优化 | Frustum + Hi-Z + MultiDrawIndirect |
| Y 轴加载 | 流式（±128 米窗口）|
| 材质系统 | 动态 Atlas（后期实现）|

本方案在**内存效率**（三态存储）、**渲染性能**（GPU 剔除 + 一次 Draw Call）、**无限扩展**（HashMap 稀疏索引）三个维度均达到目标，可支撑 128+ 区块可视距离 @ 144 FPS。
