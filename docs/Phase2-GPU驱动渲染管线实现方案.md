### 1.2 瓶颈分析（基于实际数据）

```

│                          性能瓶颈热力图                                    │
├─────────────────────────────────────────────────────────────────────────┤

```

**关键发现**：

1. **首要瓶颈已转移**：不再是 Draw Call（当前已通过 LOD 合批解决），而是 **区块调度和网格生成的同步阻塞**

2. **卸载比加载更卡顿**：
   - 加载时最低 3 FPS
   - 卸载时最低 3.3 FPS
   - 但卸载持续时间更长（~60秒 vs ~40秒）

3. **LOD 系统已在工作**：
   ```
   稳定状态 LOD 分布：LOD0=668, LOD1=1978, LOD2=3231, LOD3=3935
   峰值状态 LOD 分布：LOD0=2536, LOD1=6676, LOD2=10685, LOD3=13045
   ```

4. **CPU 三角形数量巨大**：稳态约 950万-1790万 三角形，但 GPU 渲染无压力（稳态 290 FPS）

### 1.3 为什么不再聚焦 Draw Call 优化

根据 `perf_logs` 数据：
- 当前 `triangle_count_gpu` 显示为 `N/A`，说明 GPU 指标未被正确采集
- `triangle_count_cpu` 达到 ~1790万，但 **稳态 FPS 仍有 290**
- 这意味着 **当前 Draw Call 合批已经足够好**，问题在于其他环节

### 3.2 阶段一：异步调度优化（最高优先级）

**目标**：消除区块加载/卸载时的同步阻塞

```
瓶颈分析：
┌─────────────────────────────────────────────────────────────────┐
│  加载时主线程阻塞：                                              │
│                                                                 │
│  1. fill_terrain (生成地形数据)        → 可异步                  │
│  2. generate_chunk_mesh (面剔除+网格)  → 可异步                  │
│  3. 将 Mesh 上传到 GPU                 → 可异步                  │
│  4. 更新 ChunkManager 状态              → 需同步（少量）           │
│                                                                 │
│  卸载时主线程阻塞：                                              │
│                                                                 │
│  1. 销毁 Mesh                          → 可异步                  │
│  2. 释放体素数据内存                    → 可异步                  │
│  3. 更新 ChunkManager 状态              → 需同步（少量）           │
└─────────────────────────────────────────────────────────────────┘
```

**实施方案**：

| 文件 | 操作 | 说明 |
|------|------|------|
| `src/chunk_manager.rs` | 修改 | 引入任务队列，将同步操作改为异步任务提交 |
| `src/chunk.rs` | 修改 | `fill_terrain` 返回 `Fut`，不阻塞 |
| `src/async_mesh.rs` | 扩展 | 支持 GPU 上传任务的异步调度 |

**预期收益**：

| 指标 | 当前 | 阶段一后 |
|------|------|---------|
| 加载时最低 FPS | 3 | **30-50** |
| 卸载时最低 FPS | 3.3 | **30-50** |
| 稳态 FPS | 290 | 290（不变） |

## 6. 实施建议与优先级

### 6.1 推荐的实施顺序

```
┌─────────────────────────────────────────────────────────────────┐
│                    推荐的实施路线                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Step 1: 诊断当前 Draw Call 数量                               │
│  └─ 在 `chunk_manager.rs` 添加 Draw Call 计数                   │
│  └─ 确认当前合批效果                                             │
│                                                                 │
│  Step 2: 阶段一 - 异步调度优化                                   │
│  └─ 分析 `chunk_manager.rs` 中的同步阻塞点                       │
│  └─ 引入任务队列，将加载/卸载操作异步化                           │
│  └─ **目标：将最低 FPS 从 3 提升到 30+**                         │
│                                                                 │
│  Step 3: 阶段二 - GPU 面剔除（可选）                             │
│  └─ 仅在加载时间成为问题时实施                                   │
│  └─ **目标：将加载时间从分钟级降到秒级**                           │
│                                                                 │
│  Step 4: 阶段三 - GPU 剔除（远期）                              │
│  └─ 需要 Hi-Z 遮挡剔除研究                                       │
│  └─ 预期收益有限（稳态 FPS 已够好）                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 为什么优先阶段一而非阶段二

| 对比项 | 阶段一（异步调度） | 阶段二（GPU 面剔除） |
|--------|-------------------|---------------------|
| **问题** | 加载/卸载时严重卡顿 | 新区块加载慢 |
| **发生频率** | 每次移动都触发 | 仅新区域加载 |
| **影响用户** | 持续影响体验 | 仅首次探索影响 |
| **实施复杂度** | 🟡 中 | 🔴 高 |
| **风险** | 低（纯算法优化） | 高（Bevy Compute API 成熟度） |
| **预期收益/成本** | **高** | 中 |

## 7. 回退方案

### 回退路径 A：保守优化

如果阶段一实施遇到困难，先做保守优化：

1. **分帧处理**：将大量区块的加载/卸载分到多帧
2. **优先级队列**：优先处理视野内区块，延迟视野外
3. **预加载**：在玩家移动方向提前加载

### 回退路径 B：回退到 CPU 多线程

```
┌─────────────────────────────────────────────────────────────────┐
│                    Rayon 多线程面剔除                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  use rayon::prelude::*;                                        │
│                                                                 │
│  fn generate_meshes_parallel(chunks: &[ChunkData])            │
│      -> Vec<MeshResult>                                         │
│  {                                                              │
│      chunks.par_iter()                                        │
│          .map(|chunk| generate_chunk_mesh(chunk))             │
│          .collect()                                            │
│  }                                                              │
│                                                                 │
│  预期：面剔除时间 / core_count                                  │
└─────────────────────────────────────────────────────────────────┘
```
