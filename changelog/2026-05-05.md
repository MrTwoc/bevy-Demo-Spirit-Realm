## [P0 GPU 内存泄漏修复] - 2026-05-07

### 修复

- **src/chunk_manager.rs**：`unload_distant_chunks` 和 `lru_evict` 在 despawn 实体前，现在会先从 `Assets<Mesh>` 和 `Assets<StandardMaterial>` 中移除关联的 GPU 资源
  - `ChunkEntry` 新增 `mesh_handle: Handle<Mesh>` 和 `material_handle: Handle<StandardMaterial>` 字段
  - `unload_distant_chunks` 签名变更：新增 `meshes` 和 `materials` 参数
  - `lru_evict` 签名变更：新增 `meshes` 和 `materials` 参数
  - `chunk_loader_system` 中调用上述函数时传入 `meshes` 和 `materials`

- **src/chunk.rs**：`spawn_chunk_entity` 返回值从 `Entity` 改为 `(Entity, Handle<Mesh>, Handle<StandardMaterial>)`
  - 使调用方可以将 handle 存入 `ChunkEntry`，用于卸载时清理

### 问题描述

- **P0 #1**（来自 `docs/架构总纲.md`）：`unload_distant_chunks` 和 `lru_evict` 中的 `despawn` 没有关联清理 mesh/material 资源，导致 GPU 内存泄漏
- 长时间运行（如 LRU 频繁淘汰）会导致 `Assets<Mesh>` 和 `Assets<StandardMaterial>` 中的资源持续增长

## [SubChunk 尺寸决策] - 2026-05-07

### 决策

- **SubChunk 尺寸保持 32³**，不迁移到 16×32×16
  - 16×32×16 作为备用方案记录在 `docs/架构总纲.md` §3.1.1
  - 决策依据：当前阶段（RENDER_DISTANCE=8）两种方案差异不大，32³ 代码更简洁
  - SuperChunk 合批、LOD、GPU 管线等后续方案均不依赖特定 SubChunk 尺寸
  - 未来需要 Y=±20480 超大垂直范围时可再迁移

### 文档更新

- **docs/架构总纲.md**：SubChunk 尺寸从 16×32×16 改为 32³，新增 §3.1.1 尺寸分析
- **docs/index.md**：SubChunk 分层描述从 16×32×16 改为 32³

## [P3 噪声地形 + Y轴流式加载] - 2026-05-06

### 新增

- **Cargo.toml**：`noise = "0.9"` 依赖（Simplex 噪声 + FBM 分形布朗运动）
- **src/chunk.rs**：噪声地形生成系统
  - `TERRAIN_BASE_HEIGHT = 16`：地形基准高度
  - `TERRAIN_AMPLITUDE = 32.0`：地形起伏幅度
  - `TERRAIN_SEED = 42`：全局种子，保证区块边界连续
  - `DIRT_LAYER_DEPTH = 4`：泥土层厚度
  - `SAND_HEIGHT_THRESHOLD`：低海拔区域使用沙子
  - `create_terrain_noise()`：创建 FBM 噪声生成器（4 层叠加，频率 0.005）
  - `fill_terrain()` 签名变更：新增 `coord: &ChunkCoord` 参数，使用世界坐标采样噪声

- **src/chunk_manager.rs**：Y 轴流式加载系统
  - `Y_LOAD_RADIUS = 4`：玩家上下各加载 4 层 Y 区块（±128 米）
  - `Y_UNLOAD_RADIUS = 5`：Y 轴卸载半径
  - `rebuild_load_queue()`：基于玩家 Y 坐标动态计算 Y 加载范围
  - `unload_distant_chunks()`：新增 Y 轴距离检查
  - 距离排序从 2D（dx, dz）改为 3D（dx, dy, dz）

### 修改

- **src/chunk_manager.rs**
  - 移除 `Y_LAYERS = 1` 常量（替换为 `Y_LOAD_RADIUS` + `Y_UNLOAD_RADIUS`）
  - 摄像机初始位置从 `(16, 20, 16)` 改为 `(16, 64, 16)`（地形基准高度 16 + 振幅 32 = 最高约 48）

### 地形参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 基准高度 | Y=16 | 地表平均高度 |
| 起伏幅度 | ±32 格 | 山峰最高约 Y=48，谷底最低约 Y=-16 |
| 噪声频率 | 0.005 | 较低频率 = 较大地形特征（山丘/平原） |
| 噪声层数 | 4 octaves | FBM 叠加层数，增加细节 |
| 沙滩阈值 | Y < 8 | 低海拔地表使用沙子 |

## [P2 渲染距离扩大 + 分帧加载] - 2026-05-06

### 新增

- **src/chunk_manager.rs**：分帧加载队列系统
  - `LoadedChunks` 新增 `load_queue: Vec<ChunkCoord>` 和 `last_player_chunk` 字段
  - `rebuild_load_queue()`：收集渲染距离内所有未加载区块，按距离排序
  - `CHUNKS_PER_FRAME = 4`：每帧最多加载 4 个区块，避免卡顿
  - 圆形裁剪：只加载圆形范围内的区块（`dx² + dz² ≤ RENDER_DISTANCE²`），减少角落浪费

### 修改

- **src/chunk_manager.rs**
  - `RENDER_DISTANCE`：从 2 增大到 **8**（可视距离从 64 米扩大到 256 米）
  - `UNLOAD_DISTANCE`：改为 `RENDER_DISTANCE + 1`（避免边界闪烁）
  - `setup_world()`：不再同步加载所有区块，改为将初始区块加入加载队列
  - `chunk_loader_system()`：改为分帧加载模式，玩家移动到新区块时重建队列
  - `load_chunks_around()`：已移除（功能合并到 `chunk_loader_system` + `rebuild_load_queue`）

### 性能优化

- 分帧加载：避免一帧内同步加载大量区块导致卡顿
- 距离排序：近处区块优先加载，远处区块延迟加载
- 圆形裁剪：方形区域角落的区块不再加载，减少约 21% 的区块数量

## [P1 内存泄漏修复 + 死代码清理 + 剔除验证] - 2026-05-06

### 新增

- **src/chunk_dirty.rs**：`ChunkMeshHandle` 组件，追踪每个 chunk 实体的 mesh 和 material handle

### 修改

- **src/chunk.rs**
  - `spawn_chunk_entity`：返回的实体现在包含 `ChunkMeshHandle` 组件，存储 mesh 和 material handle
  - 新增 `use crate::chunk_dirty::ChunkMeshHandle;`
  - 添加注释说明 Bevy 自动为 `Mesh3d` 实体计算 `Aabb`，启用视锥剔除和遮挡剔除

- **src/chunk_dirty.rs**
  - `rebuild_dirty_chunks`：重建 mesh 前先调用 `meshes.remove(old_mesh)` 和 `materials.remove(old_mat)` 释放旧资源
  - Query 参数从 `&Mesh3d` + `&MeshMaterial3d<StandardMaterial>` 改为 `Option<&ChunkMeshHandle>`
  - 重建后插入新的 `ChunkMeshHandle` 组件

### 剔除优化验证

- **视锥剔除（Frustum Culling）**：Bevy 自动为 `Mesh3d` 实体计算 `Aabb`，无需手动添加。当 `RENDER_DISTANCE` 增大时效果显著
- **遮挡剔除（Occlusion Culling）**：Bevy 内置支持，与 `Aabb` 配合自动工作
- **面剔除（Face Culling）**：已有，`is_face_visible()` + `ChunkNeighbors` 跨区块面剔除

### 删除

- **src/atlas.rs**：已废弃的硬编码 Atlas 槽位定义模块（项目已迁移到 `resource_pack.rs` 的动态 Atlas 系统）

### 修复

- GPU 内存泄漏：chunk mesh 重建时旧的 `Mesh` 和 `StandardMaterial` 资源现在会被正确从 `Assets` 中移除
